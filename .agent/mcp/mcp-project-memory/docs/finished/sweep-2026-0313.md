## Requirement

#### 1. DRY: Duplicated DB rebuild logic

- **What**: `server.py:_rebuild_db_from_json` (L115-153) is a near-identical copy of `build/agent.py:_rebuild_db` (both read JSON files, iterate memories, create links, set fingerprint)
- **Where**: `src/server.py`, `src/build/agent.py`
- **Why**: Changes to rebuild logic must be applied in two places or they silently diverge — a maintenance trap
- **How**: Extract a shared `rebuild_db_from_json(db, memory_store, link_store, data_dir, config)` function (in `json_store.py` or a new `db_rebuild.py`) and call it from both places. `server.py` version lacks path filtering — this is the divergence that's already happened.
- **Priority**: High
- **Effort**: Low

---

#### 2. SRP: RateLimiter bundled with OpenRouterAPI

- **What**: `openrouter.py` (380L) contains two independent classes: `OpenRouterAPI` (model info, pricing, validation) and `RateLimiter` (cross-process rate limiting with fcntl file locks)
- **Where**: `src/openrouter.py`
- **Why**: RateLimiter is a reusable concurrency primitive with no OpenRouter dependency. Bundling it makes the file harder to test and reason about independently.
- **How**: Move `RateLimiter` to `src/rate_limiter.py`. Update imports in `openrouter.py` and `build/retry.py`.
- **Priority**: Medium
- **Effort**: Low

---

#### 3. No static analysis tooling

- **What**: No linter (ruff, flake8) or type checker (mypy, pyright) is configured. No `[tool.ruff]` or `[tool.mypy]` in `pyproject.toml`.
- **Where**: `pyproject.toml`
- **Why**: Catches bugs, enforces consistency, prevents regressions. Especially valuable given the project has heavy dict/JSON manipulation with loose typing.
- **How**: Add `ruff` + `ruff format` to dev dependencies. Add `[tool.ruff]` config to `pyproject.toml`. Run as a CI gate or pre-commit step.
- **Priority**: Medium
- **Effort**: Low

- **Priority**: Medium

#### 5. Fingerprint doesn't detect content changes

- **What**: `json_store.compute_fingerprint()` hashes only filenames, not file contents. Editing a memory's JSON content won't trigger a DB rebuild on next MCP query.
- **Where**: `src/json_store.py:107-112`
- **Why**: If a user manually edits a memory JSON file (or a build updates an existing memory's summary/importance), the DB remains stale until a file is added or removed.
- **How**: Include each file's mtime or content hash in the fingerprint computation. Trade-off: content hash is slower for large corpora — mtime is a good middle ground.
- **Priority**: Medium
- **Effort**: Low

---

#### 6. `git.py:parse()` — complex method (106L, 4-level nesting)

- **What**: The `parse()` method is 106 lines with 4 levels of nesting and 3 distinct phases (header extraction, message/trailer parsing, binary diff filtering). Each phase has its own local state variables.
- **Where**: `src/git.py:72-178`
- **Why**: Hard to read, debug, and extend. Adding new commit fields requires understanding the entire method.
- **How**: Extract three private methods: `_parse_header()`, `_parse_message_and_trailers()`, `_filter_binary_diffs()`. The main `parse()` becomes a coordinator.
- **Priority**: Low
- **Effort**: Medium

---

#### 7. Missing `__init__.py` re-export for `openrouter.RateLimiter`

- **What**: `src/__init__.py` exports `LLMClient`, `BuildAgent`, etc., but `RateLimiter` is only importable via `from src.openrouter import RateLimiter`. This is inconsistent with all other public classes.
- **Where**: `src/__init__.py`
- **Why**: Inconsistency — not a bug, but surprising for consumers.
- **How**: Either add `RateLimiter` to `__all__` or (better) wait until finding #2 extracts it to its own file, then add it.
- **Priority**: Low
- **Effort**: Low
